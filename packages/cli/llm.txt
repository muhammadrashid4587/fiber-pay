fiber-pay CLI Guide (for Agents)
================================

Last updated: 2026-02-17
Fiber node target: v0.6.1
Primary RPC reference:
https://github.com/nervosnetwork/fiber/blob/v0.6.1/crates/fiber-lib/src/rpc/README.md

Purpose
-------
This file is the source-of-truth for how `@fiber-pay/cli` works. 
You, the agent, should use it to equip yourself with a p2p payment system.
Use this document for runtime behavior, command selection, output conventions, and troubleshooting.

## 🛠️ Installation
Source: `https://github.com/RetricSu/fiber-pay`
```bash
# Install & Link
git clone https://github.com/RetricSu/fiber-pay.git && cd fiber-pay
pnpm install && pnpm build
cd packages/cli && pnpm link --global
# Verify
fiber-pay --help
```

## 🛡️ Agent Policy
1. **Initialization**: Ask user: "Custom `FIBER_KEY_PASSWORD` or default?" before starting.
2. **Format**: Use `--json` for machine parsing; default output is human-readable.
3. **Safety**: Do NOT invent undocumented commands. Do NOT force env setup for single-node quickstart unless requested.
4. **Style**: Use explicit grouped commands (e.g., `fiber-pay node start`), never legacy aliases.

## ⚙️ Configuration & Environment
**Defaults**: Data Dir: `$HOME/.fiber-pay` | Network: `testnet` | RPC: `127.0.0.1:8227`
**Priority**: CLI Flags > Env Vars > Config File > Defaults

### Global Flags (all commands)
- `--profile <name>`: maps to data dir `~/.fiber-pay/profiles/<name>`
- `--data-dir <path>`: override data directory
- `--rpc-url <url>`: override RPC endpoint
- `--network <testnet|mainnet>`: override network
- `--key-password <password>`: override key password
- `--binary-path <path>`: override fnn binary path

| Env Var | Description | Default |
| :--- | :--- | :--- |
| `FIBER_DATA_DIR` | Node data storage | `$HOME/.fiber-pay` |
| `FIBER_RPC_URL` | Target Node RPC | `http://127.0.0.1:8227` |
| `FIBER_RPC_PORT` | Port for `config init` | `8227` |
| `FIBER_P2P_PORT` | Port for `config init` | `8228` |
| `FIBER_KEY_PASSWORD`| Keystore password | `fiber-pay-default-key` |
| `FIBER_NETWORK` | `mainnet` or `testnet` | `testnet` |

**Multi-Node Pattern**:
Create separate `.env` files (e.g., `.env.node-a`, `.env.node-b`) with unique `FIBER_DATA_DIR`, `FIBER_RPC_PORT`, `FIBER_P2P_PORT`.
Usage: `source .env.node-a && fiber-pay node start`

Config artifacts in this repo:
- `configs/fnn.testnet.yml`
- `configs/fnn.mainnet.yml`
- `configs/fnn.reference.yml` (full commented reference from fnn docs)

## 💻 Command Surface
*All commands support `--json` for structured output.*

### Lifecycle (`node`)
*   `start [--cors-proxy <port>] [--json]`: Start node (foreground).
*   `stop [--json]`: Stop node.
*   `status [--json]`: Unified status (process/rpc identity + diagnostics + balance).
*   `ready [--json]`: Aggregated automation readiness (node/rpc/channel/liquidity).
*   `info [--json]`: Get node ID and version details.

### Setup (`binary`, `config`)
*   `binary download [--version v0.6.1] [--force] [--json]`: Fetch fiber node binary.
*   `binary info [--json]`: Show binary path/version.
*   `config init [--data-dir <path>] [--network <net>] [--rpc-port <port>] [--p2p-port <port>] [--force] [--json]`: Initialize config.
*   `config show [--effective] [--json]`: Display current config file or effective resolved values.
*   `config get <path> [--json]`: Read a config value by path (dot + bracket syntax).
*   `config set <path> <value> [--type auto|string|number|boolean|null|json] [--json]`: Write a config value by path.
*   `config unset <path> [--json]`: Remove a config value by path.
*   `config list [--prefix <path>] [--json]`: List key paths.
*   `config profile show|set|unset`: Manage CLI-only profile keys (`binaryPath`, `keyPassword`).

### Operations (`channel`, `invoice`, `payment`)
*   **Channel**:
    *   `list [--state <state>] [--peer <id>] [--include-closed] [--json]`: List channels.
    *   `open --peer <multiaddr> --funding <CKB> [--private] [--json]`: Open channel.
    *   `close <id> [--force] [--json]`: Close channel.
    *   `abandon <id> [--json]`: Abandon channel.
    *   `update <id> [--enabled <bool>] ... [--json]`: Update TLC/fee settings.
    *   `watch [--json] [--on-timeout fail|success]`: Monitor channel state changes.
*   **Invoice**:
    *   `create --amount <CKB> --description <txt> [--expiry <min>] [--json]`: Gen invoice.
    *   `parse <invoiceString> [--json]`: Parse invoice.
    *   `get <paymentHash> [--json]`: Query invoice status.
    *   `cancel <paymentHash> [--json]`: Cancel invoice.
    *   `settle <paymentHash> --preimage <hex> [--json]`: Manually settle.
*   **Payment**:
    *   `send <invoice> | --to <nodeId> --amount <CKB> [--json]`: Pay an invoice.
    *   `get <paymentHash> [--json]`: Check payment status.
    *   `watch <paymentHash> [--json] [--until SUCCESS|FAILED|TERMINAL] [--on-timeout fail|success]`: Monitor payment outcome.

### Network & Info (`peer`)
*   `peer list [--json]`: Show connected peers.
*   `peer connect <multiaddr> [--json]`: Connect to peer.
*   `peer disconnect <peerId> [--json]`: Disconnect.

## 📦 JSON Contract (vNext)
- Non-stream commands in `--json` mode return one envelope:
    - success: `{"success":true,"data":{...}}`
    - failure: `{"success":false,"error":{"code":"...","message":"...","recoverable":true|false?,"suggestion":"..."?,"details":{...}?}}`
- `payment send` returns `success: true` when RPC accepted the request, with `data.status` in `success|pending|failed`.
- In `--json` mode, command stdout is machine-only JSON (no progress bars, emoji, or human prose).

### Error object fields
- `code` (required): stable machine code
- `message` (required): human-readable failure summary
- `recoverable` (optional): whether agent can usually retry after correction
- `suggestion` (optional): immediate remediation hint
- `details` (optional): structured context (provided/expected/timeouts/etc)

## 🌊 Streaming Contract (`watch --json`)
- Streaming commands emit NDJSON (one JSON object per line), not a single pretty JSON blob.
- `channel watch --json` events:
    - `snapshot`: current channels + summary
    - `state_change`: transition for one channel
    - `terminal`: exit condition reached
- `payment watch --json` events:
    - `status_transition`
    - `terminal`
- `node start --json` lifecycle events:
    - `startup_stage` with `data.stage` in `init|binary_resolved|key_initialized|process_started|rpc_ready|bootnodes_connected|startup_complete`
    - `node_started`
    - `node_stopped`
- Timeout behavior is configurable via `--on-timeout fail|success`.
- Timeout/errors use failure envelope only when timeout behavior is `fail`.

## 🚀 Common Workflows

**1. Quick Start (Single Node)**
```bash
fiber-pay --profile local-a binary download
# foreground process (keep this terminal open)
fiber-pay --profile local-a node start
# In new terminal:
fiber-pay --profile local-a node status
```

## ✅ Agent 快速验证（复制粘贴）
```bash
fiber-pay binary info --json
fiber-pay node status --json
fiber-pay node ready --json
fiber-pay channel list --include-closed --json
```

## ⚠️ 当前限制（Agent 须知）
- 通道开户到可用需要时间；`open` 成功不代表立即 `ChannelReady`。
- 长时命令（如 `node start`、`watch`）是阻塞式前台行为，自动化中需放在独立进程/终端。
- `watch --json` 为 NDJSON 流式输出，不是单个 JSON 文档。

## 🧭 错误码参考（节选）
| Code | Meaning | Agent Action |
| :--- | :--- | :--- |
| `NODE_ALREADY_RUNNING` | 节点已在运行 | 跳过启动或先停止旧进程 |
| `NODE_NOT_RUNNING` | 节点未运行/陈旧 PID | 先 `node start` 或清理运行态 |
| `NODE_RPC_NOT_READY` | 启动后 RPC 未就绪 | 延迟重试并检查配置/端口 |
| `CHANNEL_NOT_FOUND` | 指定通道不存在 | 先 `channel list --include-closed --json` 再操作 |
| `PAYMENT_SEND_INPUT_INVALID` | 支付参数不合法 | 修正 `invoice` 或 `--to/--amount` |
| `PAYMENT_WATCH_TIMEOUT` | 支付未在超时内终态 | 继续轮询 `payment get` 或延长超时 |
| `CONFIG_NOT_FOUND` | 配置文件不存在 | 先 `config init --network testnet` |

完整机器可读清单：`packages/cli/error-codes.json`

**2. Receive Payment**
```bash
fiber-pay invoice create --amount 10 --description "Test"
# Returns <invoice_string> and <payment_hash>
fiber-pay invoice get <payment_hash>
```

**3. Send Payment**
```bash
fiber-pay payment send <invoice_string>
fiber-pay payment watch <payment_hash>
```

## 🔧 Troubleshooting
*   **RPC Error**: Check `fiber-pay node status`. Verify `FIBER_RPC_URL` matches the running node.
*   **Browser Access**: Must run with `fiber-pay node start --cors-proxy`.
*   **Apple Silicon**: If x86_64 fallback used, run `softwareupdate --install-rosetta`.
