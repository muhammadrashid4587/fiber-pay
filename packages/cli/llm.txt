fiber-pay CLI Guide (for Agents)
================================

Last updated: 2026-02-12
Fiber node target: v0.6.1
Primary RPC reference:
https://github.com/nervosnetwork/fiber/blob/v0.6.1/crates/fiber-lib/src/rpc/README.md

Purpose
-------
This file is the source-of-truth for how `@fiber-pay/cli` works and how an agent should use it.
Use this document for runtime behavior, command selection, output conventions, and troubleshooting.

CLI Mental Model
----------------
`fiber-pay` has grouped commands:
- `node`     lifecycle / process state
- `channel`  channel lifecycle + status
- `invoice`  invoice lifecycle + status
- `payment`  payment lifecycle + status
- `peer`     peer connectivity
- `binary`   fnn binary download/info
- `balance`  balance summary

Most business commands are RPC-driven and require a running Fiber node.

Runtime Behavior (Important Defaults)
-------------------------------------
Node start behavior:
- `fiber-pay node start` starts Fiber node in foreground.
- Bootnode auto-connect is enabled: after RPC readiness, bootnodes from config are connected automatically.
- CORS proxy is optional, not default:
  - `fiber-pay node start --cors-proxy` => starts proxy on `28227`
  - `fiber-pay node start --cors-proxy 3000` => starts proxy on custom port
- On shutdown (`Ctrl+C`), node and optional CORS proxy are stopped.

RPC readiness behavior:
- RPC commands wait for node readiness via SDK client helper.
- If node is not ready/running, command fails with an actionable error.

Output Conventions
------------------
- Default output: human-readable (operator-friendly).
- `--json`: machine-readable payload for automation.
- Fatal path with `--json`: structured `{ success: false, error: ... }` output.

Agent rule:
- If output will be parsed downstream, always pass `--json`.
- For interactive troubleshooting, prefer default human output.

Core Workflows
--------------
1) Bootstrap node
- `fiber-pay binary download`
- `fiber-pay node start`
- `fiber-pay node status`
- `fiber-pay node info --json`

2) Receive payment (invoice flow)
- `fiber-pay invoice create --amount 10 --description "service"`
- `fiber-pay invoice get <paymentHash>`
- Optional: `fiber-pay invoice cancel <paymentHash>`

3) Send payment
- `fiber-pay payment send <invoice>`
- `fiber-pay payment get <paymentHash>`
- `fiber-pay payment watch <paymentHash>`

4) Channel operations
- `fiber-pay channel list`
- `fiber-pay channel get <channelId>`
- `fiber-pay channel open --peer <peerIdOrMultiaddr> --funding <CKB>`
- `fiber-pay channel close <channelId>`

5) Connectivity + liquidity checks
- `fiber-pay peer list`
- `fiber-pay balance`

Command Surface (Reference)
---------------------------
Node:
- fiber-pay node start [--cors-proxy [port]]
- fiber-pay node stop
- fiber-pay node status
- fiber-pay node info [--json]

Channel:
- fiber-pay channel list [--state <state>] [--peer <peerId>] [--include-closed] [--raw] [--json]
- fiber-pay channel get <channelId> [--raw] [--json]
- fiber-pay channel watch [--interval <sec>] [--timeout <sec>] [--channel <id>] [--peer <id>] [--state <state>] [--until <state>] [--include-closed] [--no-clear] [--json]
- fiber-pay channel open --peer <peerIdOrMultiaddr> --funding <CKB> [--private]
- fiber-pay channel close <channelId> [--force]
- fiber-pay channel abandon <channelId>
- fiber-pay channel update <channelId> [--enabled <true|false>] [--tlc-expiry-delta <ms>] [--tlc-minimum-value <hex>] [--tlc-fee-proportional-millionths <value>]

Invoice:
- fiber-pay invoice create [amount] [--amount <CKB>] [--description <text>] [--expiry <minutes>]
- fiber-pay invoice get <paymentHash> [--json]
- fiber-pay invoice parse <invoiceString>
- fiber-pay invoice cancel <paymentHash> [--json]
- fiber-pay invoice settle <paymentHash> --preimage <0x...> [--json]

Payment:
- fiber-pay payment send [invoice] [--invoice <invoice>] [--to <nodeId>] [--amount <CKB>] [--max-fee <CKB>] [--json]
- fiber-pay payment get <paymentHash> [--json]
- fiber-pay payment watch <paymentHash> [--interval <sec>] [--timeout <sec>] [--json]

Peer:
- fiber-pay peer list [--json]
- fiber-pay peer connect <multiaddr> [--json]
- fiber-pay peer disconnect <peerId> [--json]

Binary:
- fiber-pay binary download [--version <vX.Y.Z>] [--force] [--json]
- fiber-pay binary info [--json]

Convenience:
- fiber-pay balance [--json]

Command Style Policy
--------------------
- Grouped commands are the only supported CLI style.
- Legacy top-level aliases are intentionally removed to avoid ambiguous usage.
- Use explicit grouped forms only (for example: `fiber-pay node start`, `fiber-pay binary info`).

Environment Variables
---------------------
- FIBER_BINARY_PATH
- FIBER_DATA_DIR
- FIBER_NETWORK (`testnet` | `mainnet`)
- FIBER_RPC_URL
- FIBER_KEY_PASSWORD

Troubleshooting (Agent Checklist)
---------------------------------
1) "RPC not responding" or request timeout
- Verify process: `fiber-pay node status`
- Ensure endpoint: check `FIBER_RPC_URL`
- Restart node if needed: `fiber-pay node stop && fiber-pay node start`

2) Browser cannot call RPC
- Start with proxy: `fiber-pay node start --cors-proxy`
- Use proxy URL (`http://127.0.0.1:28227` by default)

3) Binary missing or not executable
- `fiber-pay binary info`
- `fiber-pay binary download --force`

4) Apple Silicon download fallback issues
- If x86_64 fallback is selected, Rosetta 2 is required:
  `softwareupdate --install-rosetta --agree-to-license`

Maintenance Notes (for CLI contributors)
----------------------------------------
Relevant code layout:
- src/index.ts (root command/error handling)
- src/commands/* (command groups)
- src/lib/* (shared config/rpc/format/pid/bootnode)

When changing CLI behavior:
1) Keep grouped command semantics stable.
2) Keep output convention stable (human default + `--json`).
3) Do not add legacy-style top-level aliases.
4) Validate:
   - pnpm --filter @fiber-pay/cli typecheck
   - pnpm --filter @fiber-pay/cli build
5) Update this file when command behavior or guidance changes.
