fiber-pay Guide (for Agents)
============================

Last updated: 2026-02-18
Fiber node target: v0.6.1
Primary RPC reference:
https://github.com/nervosnetwork/fiber/blob/v0.6.1/crates/fiber-lib/src/rpc/README.md

Purpose
-------
This file is the source-of-truth for how fiber-pay works.
You, the agent, should use it to equip yourself with a CKB p2p payment system.
Use this document for integration tiers, command selection, output conventions,
runtime orchestration, and troubleshooting.

## Integration Tiers (pick one)

| Tier | Interface | When to use |
| :--- | :--- | :--- |
| **1. CLI** | `@fiber-pay/cli` | Shell-based agent or human operator |
| **2. Runtime REST API** | (runtime proxy) | Custom HTTP client, browser, or other languages |

Both tiers share the same underlying runtime job orchestration.
The CLI auto-routes through the REST API when the runtime proxy is active.

## Tier 1 — CLI (`@fiber-pay/cli`)

### Install
```bash
git clone https://github.com/RetricSu/fiber-pay.git && cd fiber-pay
pnpm install && pnpm build
cd packages/cli && pnpm link --global
fiber-pay --help
```

### Agent Policy
1. Ask user: "Custom `FIBER_KEY_PASSWORD` or default?" before starting.
2. Always use `--json` for machine parsing.
3. Do NOT invent undocumented commands.
4. Use explicit grouped commands (e.g., `fiber-pay node start`), never legacy aliases.
5. Be proactive: if environment is not ready, execute setup steps end-to-end instead of waiting for user to request each command.
6. Before first payment attempt, ask user for a funding address to top up channel liquidity (or confirm already funded), then verify readiness again.

### Agent Autonomy Playbook (default behavior)
Use this workflow unless the user explicitly asks for manual step-by-step control.

1. **Bootstrap automatically**
   - `fiber-pay binary info --json`; if missing/outdated, run `fiber-pay binary download --version v0.6.1 --json`.
   - `fiber-pay config show --effective --json`; if missing, run `fiber-pay config init --network testnet --json`.
   - `fiber-pay node status --json`; if not running, run `fiber-pay node start --json` (or profile-scoped equivalent).

2. **Funding checkpoint (required for payments)**
   - Ask user for funding source and destination address context:
     - "Please provide the funding address/source (faucet or wallet) and target node/profile to fund."
   - After user confirms funding action is done, re-check:
     - `fiber-pay node status --json`
     - `fiber-pay node ready --json`
     - `fiber-pay channel list --include-closed --json`
   - If liquidity/channel readiness is not sufficient, explain concrete next action (open channel / add funds) and continue verification loop.

3. **Payment readiness gate**
   - Only proceed to `payment send` when:
     - node process is running,
     - RPC is reachable,
     - runtime proxy is available,
     - at least one usable channel/liquidity path exists.
   - Preferred preflight command set:
     - `fiber-pay node ready --json`
     - `fiber-pay runtime status --json`
     - `fiber-pay channel list --state ChannelReady --json`

4. **Execution discipline**
   - For payment execution use `fiber-pay payment send ... --wait --json`.
   - On failure, classify by `error.code`, apply the documented recovery, then retry with bounded attempts.
   - Summarize outcome with job/payment identifiers (`jobId`, `paymentHash`, terminal state).

### Configuration & Environment
**Defaults**: Data Dir: `$HOME/.fiber-pay` | Network: `testnet` | RPC: `127.0.0.1:8227`
**Priority**: CLI Flags > Env Vars > Config File > Defaults

#### Global Flags (all commands)
- `--profile <name>`: maps to data dir `~/.fiber-pay/profiles/<name>`
- `--data-dir <path>`: override data directory
- `--rpc-url <url>`: override RPC endpoint
- `--network <testnet|mainnet>`: override network
- `--key-password <password>`: override key password
- `--binary-path <path>`: override fnn binary path

| Env Var | Description | Default |
| :--- | :--- | :--- |
| `FIBER_DATA_DIR` | Node data storage | `$HOME/.fiber-pay` |
| `FIBER_RPC_URL` | Target Node RPC | `http://127.0.0.1:8227` |
| `FIBER_RPC_PORT` | Port for `config init` | `8227` |
| `FIBER_P2P_PORT` | Port for `config init` | `8228` |
| `FIBER_KEY_PASSWORD`| Keystore password | `fiber-pay-default-key` |
| `FIBER_NETWORK` | `mainnet` or `testnet` | `testnet` |

**Multi-Node Pattern (`--profile`)**:
Use one profile per node. `--profile <name>` maps to `~/.fiber-pay/profiles/<name>`, giving each node isolated config/state.

```bash
fiber-pay --profile rt-a binary download --version v0.6.1 --json
fiber-pay --profile rt-b binary download --version v0.6.1 --json

fiber-pay --profile rt-a node start --runtime-proxy-listen 127.0.0.1:9729 --json
fiber-pay --profile rt-b node start --runtime-proxy-listen 127.0.0.1:9829 --json

fiber-pay --profile rt-a node status --json
fiber-pay --profile rt-b node status --json
```

Prefer profile-scoped commands over ad-hoc `source .env.*` switching.

Config files in this repo:
- `configs/fnn.testnet.yml`
- `configs/fnn.mainnet.yml`
- `configs/fnn.reference.yml` (full commented reference)

### Command Surface
*All commands support `--json` for structured output.*

#### Lifecycle (`node`)
*   `start [--runtime-daemon] [--runtime-proxy-listen <host:port>] [--json]`: Start node (foreground) with runtime watcher + proxy; `--runtime-daemon` runs watcher detached.
*   `stop [--json]`: Stop node (and runtime daemon if running).
*   `status [--json]`: Unified status (process/rpc identity + diagnostics + balance).
*   `ready [--json]`: Aggregated automation readiness (node/rpc/channel/liquidity).
*   `info [--json]`: Get node ID and version.

#### Runtime Monitor (`runtime`)
*   `start [--daemon] [--proxy-listen <host:port>] [--channel-poll-ms <ms>] [--invoice-poll-ms <ms>] [--payment-poll-ms <ms>] [--peer-poll-ms <ms>] [--health-poll-ms <ms>] [--include-closed <bool>] [--completed-ttl-seconds <seconds>] [--state-file <path>] [--flush-ms <ms>] [--webhook <url>] [--websocket <host:port>] [--log-min-priority <critical|high|medium|low>] [--log-type <comma-separated>] [--json]`: Start polling monitor + HTTP proxy.
*   `status [--json]`: Runtime process & proxy health.
*   `stop [--json]`: Stop runtime process.

#### Job Orchestration (`job`)
*Requires runtime proxy. Auto-started by `node start` or `runtime start`.*
*   `job list [--state <state>] [--type <payment|invoice|channel>] [--limit <n>] [--offset <n>] [--json]`: List runtime jobs.
*   `job get <jobId> [--json]`: Get job details (state, result, retries, error).
*   `job cancel <jobId> [--json]`: Cancel a running job.
*   `job events <jobId> [--json]`: Show state transition history for a job.

#### Setup (`binary`, `config`)
*   `binary download [--version v0.6.1] [--force] [--json]`: Fetch fiber node binary.
*   `binary info [--json]`: Show binary path/version.
*   `config init [--data-dir <path>] [--network <net>] [--rpc-port <port>] [--p2p-port <port>] [--force] [--json]`: Initialize config.
*   `config show [--effective] [--json]`: Display config.
*   `config get <path> [--json]`: Read config value (dot + bracket syntax).
*   `config set <path> <value> [--type auto|string|number|boolean|null|json] [--json]`: Write config value.
*   `config unset <path> [--json]`: Remove config value.
*   `config list [--prefix <path>] [--json]`: List key paths.
*   `config profile show|set|unset`: Manage profile keys (`binaryPath`, `keyPassword`).

#### Operations (`channel`, `invoice`, `payment`)
*   **Channel**:
    *   `list [--state <state>] [--peer <id>] [--include-closed] [--json]`
    *   `open --peer <multiaddr> --funding <CKB> [--private] [--json]`
    *   `close <id> [--force] [--json]`
    *   `abandon <id> [--json]`
    *   `update <id> [--enabled <bool>] ... [--json]`
    *   `watch [--json] [--on-timeout fail|success]`
*   **Invoice**:
    *   `create --amount <CKB> --description <txt> [--expiry <min>] [--json]`
    *   `parse <invoiceString> [--json]`
    *   `get <paymentHash> [--json]`
    *   `cancel <paymentHash> [--json]`
    *   `settle <paymentHash> --preimage <hex> [--json]`
*   **Payment**:
    *   `send <invoice> | --to <nodeId> --amount <CKB> [--wait] [--timeout <seconds>] [--json]`: Pay an invoice. When runtime proxy is active, automatically creates a runtime job. `--wait` blocks until the job reaches a terminal state.
    *   `get <paymentHash> [--json]`
    *   `watch <paymentHash> [--json] [--until SUCCESS|FAILED|TERMINAL] [--on-timeout fail|success]`
    *   `route --hops <pubkeys> [--amount <CKB>] [--json]`: Build a payment route.
    *   `send-route --router <json> [--invoice <inv>] [--payment-hash <hash>] [--keysend] [--dry-run] [--json]`: Send via pre-built route.

#### Network Graph (`graph`, `peer`)
*   `graph nodes [--limit <n>] [--after <cursor>] [--json]`: List network graph nodes.
*   `graph channels [--limit <n>] [--after <cursor>] [--json]`: List network graph channels.
*   `peer list [--json]`: Connected peers.
*   `peer connect <multiaddr> [--json]`: Connect to peer.
*   `peer disconnect <peerId> [--json]`: Disconnect.

### JSON Contract
Non-stream commands in `--json` return one envelope:
- success: `{"success":true,"data":{...}}`
- failure: `{"success":false,"error":{"code":"...","message":"...","recoverable":true|false,"suggestion":"...","details":{...}}}`

`payment send --json` returns `success: true` when accepted, with `data.status` in `success|pending|failed`.
When `--wait` is used and runtime proxy is active, the response includes `data.jobId` and `data.jobState`.

Error object fields:
- `code` (required): stable machine code
- `message` (required): human-readable failure summary
- `recoverable` (optional): whether agent can retry after correction
- `suggestion` (optional): remediation hint
- `details` (optional): structured context

### Streaming Contract (`watch --json`)
Streaming commands emit NDJSON (one JSON object per line).
- `channel watch --json`: `snapshot` → `state_change` → `terminal`
- `payment watch --json`: `status_transition` → `terminal`
- `node start --json`: `startup_stage` (init|binary_resolved|key_initialized|process_started|rpc_ready|bootnodes_connected|startup_complete) → `node_started` → `node_stopped`
- `runtime start --json`: `runtime_starting` → `runtime_started` → `runtime_alert` → `runtime_stopping` → `runtime_stopped`
- `--on-timeout fail|success` controls exit behavior on timeout.

## Tier 2 — Runtime Proxy REST API

The runtime proxy starts automatically with `node start` (default `127.0.0.1:8229`).
It is a plain HTTP/JSON API — no auth, no TLS. All below paths are relative to the proxy base URL.

### Job submission (POST)
```
POST /jobs/payment   { "params": { "invoice": "...", "sendPaymentParams": {...} }, "options": { "idempotencyKey": "...", "maxRetries": 3 } }
POST /jobs/invoice   { "params": { "newInvoiceParams": {...} }, "options": {...} }
POST /jobs/channel   { "params": { "openChannelParams": {...} }, "options": {...} }
```
Response: full job record `{ id, type, state, params, result, error, ... }`

### Job management (GET / DELETE)
```
GET    /jobs                          ?state=pending&type=payment&limit=10&offset=0
GET    /jobs/:id
GET    /jobs/:id/events
DELETE /jobs/:id                      → 204 on success (cancel)
```

### Monitoring (GET)
```
GET /monitor/status                   → runtime health + uptime
GET /monitor/list_alerts              ?limit=50&min_priority=high&type=...&source=...
GET /monitor/list_tracked_invoices    → currently-tracked invoices
GET /monitor/list_tracked_payments    → currently-tracked payments
```

### Job states
`pending` → `running` → `succeeded` | `failed` | `cancelled`

Jobs have automatic retry with exponential backoff. Terminal states are permanent.

## Common Workflows

**1. Quick Start (Single Node)**
```bash
fiber-pay --profile local-a binary download
fiber-pay --profile local-a node start          # foreground, keep terminal open
# In new terminal:
fiber-pay --profile local-a node status --json
```

**2. Send Payment (CLI, runtime-backed)**
```bash
fiber-pay payment send <invoice> --wait --json
# Returns job status with paymentHash, jobId, jobState
```

**3. Receive Payment**
```bash
fiber-pay invoice create --amount 10 --description "Test" --json
fiber-pay invoice get <payment_hash> --json
```

**4. Monitor Jobs**
```bash
fiber-pay job list --state pending --json
fiber-pay job get <jobId> --json
fiber-pay job events <jobId> --json
```

```bash
fiber-pay binary info --json

fiber-pay node ready --json

```

## Limitations
- Channel open → usable takes time; `open` success ≠ immediate `ChannelReady`.
  Use `waitForChannelReady()` (SDK) or `channel watch --json` (CLI).
- Long-running commands (`node start`, `watch`) are blocking foreground processes;
  run in a dedicated terminal or use `--runtime-daemon`.
- `watch --json` emits NDJSON stream, not a single JSON document.
- Runtime proxy has no auth/TLS — bind to localhost only in production.

## Error Code Reference
| Code | Meaning | Agent Action |
| :--- | :--- | :--- |
| `NODE_ALREADY_RUNNING` | Node already running | Skip start or stop first |
| `NODE_NOT_RUNNING` | Node not running / stale PID | `node start` or clean up |
| `NODE_RPC_NOT_READY` | RPC not ready after start | Retry with delay, check port |
| `CHANNEL_NOT_FOUND` | Channel does not exist | `channel list --include-closed --json` first |
| `PAYMENT_SEND_INPUT_INVALID` | Bad payment params | Fix `invoice` or `--to/--amount` |
| `PAYMENT_WATCH_TIMEOUT` | Payment not terminal in time | Continue polling or extend timeout |
| `CONFIG_NOT_FOUND` | Config file missing | `config init --network testnet` |
| `RUNTIME_PROXY_REQUIRED` | Job command needs runtime proxy | `runtime start` or `node start` first |
| `JOB_CANCEL_FAILED` | Job cancel failed | Check job ID and state |
| `JOB_EVENTS_FAILED` | Job events fetch failed | Check job ID exists |

Full machine-readable list: `packages/cli/error-codes.json`

## Troubleshooting
*   **RPC Error**: `fiber-pay node status --json`. Verify `FIBER_RPC_URL` matches the running node.
*   **Runtime proxy not reachable**: `fiber-pay runtime status --json`. Ensure proxy is started.
*   **Job stuck in pending**: Check `fiber-pay job events <id> --json` for transition history and errors.
*   **Apple Silicon**: If x86_64 fallback used, `softwareupdate --install-rosetta`.
